# Wave 3 — overlays & positioning: tooltip, popover, scroll-area, select, dropdown-menu

Read `00-overview.md` first. This is the heaviest wave (Select alone has ~58 consumer files, and asChild→render lands on Tooltip/Popover/DropdownMenu triggers). Consider doing it in sub-steps and running `tsc` between each.

## Components (ui/)
`tooltip`, `popover`, `scroll-area`, `select`, `dropdown-menu`

```bash
npx shadcn@latest add tooltip popover scroll-area select dropdown-menu --overwrite --diff   # review first
npx shadcn@latest add tooltip popover scroll-area select dropdown-menu --overwrite
npx prettier --write src/components/ui/{tooltip,popover,scroll-area,select,dropdown-menu}.tsx
```

## Consumer breaks to expect

### 1. asChild → render (biggest volume) — see overview for the rule
Triggers in this wave (counts across `src/` + `registry/`):
- `<DropdownMenuTrigger asChild>` ~35
- `<PopoverTrigger asChild>` ~32
- `<TooltipTrigger asChild>` ~28

Sweep: `grep -rln 'DropdownMenuTrigger\|PopoverTrigger\|TooltipTrigger' src/ registry/` then convert every `asChild` on these. Watch for non-button render targets (add `nativeButton={false}`).

### 2. Select — the largest API change (~58 files use `<SelectTrigger>`/`<SelectValue>`)
Read `.agents/skills/shadcn/rules/base-vs-radix.md` §Select thoroughly, and `npx shadcn@latest docs select`. Base differences vs Radix:
- **Placeholder:** Radix `<SelectValue placeholder="Pick one" />`. Base has no `placeholder` on `SelectValue`; the empty state is represented by an item with `value={null}` in the `items` array, or handled via the generated wrapper. **Check what the regenerated `src/components/ui/select.tsx` actually expects** before mass-editing — the shadcn Base wrapper may accept a `placeholder` prop it forwards. Do not guess; open the file.
- **`items` prop on root:** Base's underlying primitive wants an `items` array for value/placeholder rendering. Confirm whether the shadcn wrapper makes this optional. If required, every `<Select>` needs an `items={...}` derived from its `<SelectItem>`s.
- **Positioning:** Radix `<SelectContent position="popper">` → Base uses `alignItemWithTrigger`. Only `registry/patterns/select/select-15.tsx` uses `position=` here.
- **Controlled value:** Radix single-select string value/onValueChange. Base supports `multiple` + object values; single-select still string. Most consumers are simple string selects and should keep working, but verify onValueChange signatures type-check.
- **CSS var:** `--radix-select-trigger-width` → `--anchor-width` (grep to confirm none present; the width vars found were on dropdown/popover, see below).

Select consumers are concentrated in: `registry/blocks/*/editor/fields.tsx` (many editor field selects — often CTA `variant` selects), `registry/patterns/select/select-01..15.tsx`, `registry/forms/**`, `registry/intents/**`. Let `tsc` drive the exact list.

### 3. `--radix-*-trigger-width` → `--anchor-width` (popover/dropdown width matching)
These files use the Radix width var in a `className` (`w-(--radix-...-width)`):
- `registry/forms/react-hook-form/fields/rhf-fields-08.tsx` (`--radix-popover-trigger-width`)
- `registry/forms/react-hook-form/fields/rhf-fields-09.tsx` (`--radix-dropdown-menu-trigger-width`)
- `registry/forms/tanstack-form/fields/tsf-fields-08.tsx`
- `registry/forms/tanstack-form/fields/tsf-fields-09.tsx`
- `registry/patterns/command/command-12.tsx`, `command-13.tsx`, `command-14.tsx` (`--radix-popover-trigger-width`)
- `registry/patterns/dropdown/dropdown-18.tsx`, `dropdown-19.tsx` (`--radix-dropdown-menu-trigger-width`)
- also inside `src/components/ui/dropdown-menu.tsx` and `src/components/ui/select.tsx` — but those get regenerated by the overwrite, so check the regenerated versions use the Base var.

Replace `w-(--radix-popover-trigger-width)` and `w-(--radix-dropdown-menu-trigger-width)` with `w-(--anchor-width)`. **Verify** the Base var name against the regenerated ui files and `node_modules/@base-ui/react` (`--anchor-width` confirmed present). Note: Base exposes the anchor width on the **Positioner**; make sure the class is on an element inside the popup where that var is in scope (mirror whatever the regenerated `select.tsx`/`dropdown-menu.tsx` do).

### 4. State-attr renames in classNames
Grep consumers of these components for `data-[state=open]` / `data-[state=closed]` and rename to `data-open` / `data-closed`. Verify which element carries the attr per overview (read `node_modules/@base-ui/react/<comp>/**/*DataAttributes.d.ts`). Tooltip/Popover/DropdownMenu popups emit `data-open`/`data-closed`; triggers may emit `data-popup-open` — confirm before rewriting `group-data-*`.

### 5. Direct primitive imports
Grep `registry/` for `from 'radix-ui'` after migrating — if any tooltip/popover/select/dropdown/scroll-area pattern imports the primitive directly, hand-migrate its import + API (mirror the regenerated ui file's structure). (Wave 2 had accordion-06/07 and switch-07 like this.)

## Behavior changes to flag (verify visually)
- Tooltip open/close delay and positioning defaults differ slightly between Radix and Base — confirm tooltips still appear on hover/focus.
- Select dropdown alignment: `alignItemWithTrigger` (Base default true) makes the selected item align over the trigger, which looks different from Radix `popper`. Decide per pattern whether to set `alignItemWithTrigger={false}`.
- Popover/Dropdown width-matching relies on `--anchor-width` being in scope — check the width actually matches the trigger.

## Done when
`npx tsc --noEmit` = 0 errors, `pnpm build` completes, no `radix-ui`/`@radix-ui` imports remain for these 5 components in `src/` or `registry/`. Write `.migration/wave-3.md`, commit.
